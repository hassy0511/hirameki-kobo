# カリキュラム拡張フレームワーク

## 基本単位

ひらめき工房のカリキュラムは、次の4階層で管理する。

1. 学年：小学1年生、小学2年生など
2. 学習ライン：数と計算、計測、図形、データ活用などのまとまり
3. 修理ステージ：8問で一つの具体的技能を確認する単位
4. 問題タイプ：具体物、図、式、穴埋め、文章題、操作問題

学習ラインは工房の設備として並び、別の島や別世界として扱わない。

## ステージの粒度

教科書の「単元」をそのまま1ステージにはしない。複数社の教科書単元から共通技能を抽出し、次の条件を満たす大きさへ分解する。

- 1ステージの目標を一文で説明できる
- 8問の中で数値や表現を変えられる
- 正誤から、どの技能でつまずいたか判断できる
- 前提技能と次の技能を明示できる

## 共通の進み方

1. 具体物を見たり操作したりする
2. まとまり、数直線、図などの視覚補助を使う
3. 数や記号で表す
4. 補助を減らして解く
5. 形式を混ぜて理解を確認する
6. 短い文章や日常場面へ応用する

## 工房内の配置

各ラインは原則11ステージとし、3区画に配置する。

- A 基礎作業台：ステージ1〜4
- B 組み替えフロア：ステージ5〜8
- C 検査室：ステージ9〜11

この11ステージ構造は画面上の一貫性のための標準であり、教科内容を無理に分割・結合してはならない。必要な場合はステージ数を変更する。

## カリキュラムと演出の分離

学習目標、数値範囲、前提技能、難易度はカリキュラムファイルで管理する。キャラクター、会話、装置名、色、報酬とは分ける。

各技能には `canonicalSkillId` を付ける。同じ技能が複数ラインで練習されても、保護者向け記録では一つの技能として集約できるようにする。

学年を追加するときは `CURRICULA.g1`、`CURRICULA.g2` のように学年階層を先に置く。問題、進捗、履歴、最近問、タイムアタックは必ず `gradeId` と組にして保存し、同じライン名を学年間で使っても記録が混ざらないようにする。

IDは次の規則にする。

- 学年ID：`g1`、`g2`
- ステージID：既存G1は維持、新規G2は `g2_` で始める
- 技能ID：`g1.number.to100`、`g2.number.to1000` のように学年を含める
- 画面上の順番は配列index、保存と参照は安定したstage IDを使う

## 問題生成とやり込み

- 通常8問は操作を中心にし、文章・場面問題と確認問題を混ぜる
- semantic signatureを使い、同一プレイと直近プレイの重複を避ける
- seedを履歴へ保存し、同じ条件の問題を調査可能にする
- ライン完走後は、そのラインの代表技能を混ぜた12問制タイムアタックを解放する
- タイムアタックの自己ベストは学年・ライン単位で独立して保存する

## 新しいラインの条件

- 学習指導要領と現行教科書の対応が確認できる
- 既存ラインの色替えではなく、内容に合う操作がある
- 少なくとも一つ、その技能だから成立する専用装置がある
- ダジャレや長い物語がなくても目的が伝わる
- 工房コントロール室、設計図、修理ベンチ、レポート、図鑑へ接続できる
